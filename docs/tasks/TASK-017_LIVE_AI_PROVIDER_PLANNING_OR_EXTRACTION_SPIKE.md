# TASK-017: Live AI Provider Planning Or Extraction Spike

Status: proposed
Owner/Agent: unassigned
Context shards: `security.md`, `business-understanding.md`, `testing.md`

## Goal

Plan the first safe live AI provider path for FlowAI so the owner-first builder can improve business understanding without allowing AI to bypass sourceRefs, validation, deterministic fallbacks, or Workflow JSON safety.

This task is planning/spec only unless explicitly upgraded to an implementation task.

## Scope

- Define a backend-only live provider integration boundary.
- Define how local development secrets may be loaded without committing or exposing them.
- Define model-selection policy without hardcoding a single future model.
- Define the first allowed live AI use case: source-backed BusinessUnderstanding extraction/refinement.
- Define structured provider output requirements.
- Define fallback behavior when provider output is invalid, unsafe, unavailable, or rate-limited.
- Define redaction and logging rules.
- Define non-CI smoke test expectations.
- Define the next implementation split.

## Non-Goals

- No live OpenAI call in this planning task.
- No OpenAI SDK dependency in this planning task.
- No `.flowai.local.json` application read in this planning task.
- No provider credentials in git, workflow JSON, BusinessUnderstanding, traces, logs, tests, or docs examples.
- No AI-generated final Workflow JSON.
- No upload endpoint.
- No PDF/DOCX parser.
- No crawling.
- No RAG, embeddings, or vector DB.
- No persistence/database.
- No auth, tenants, or billing.
- No live Telegram or WhatsApp.
- No production webhook behavior.
- No Workflow DSL or runtime-core semantic change.

## Recommended First Live AI Use Case

Start with source-backed BusinessUnderstanding refinement only:

```text
SourceDocument / owner text
-> deterministic fallback understanding
-> live provider structured extraction/refinement
-> schema validation
-> sourceRef and safety validation
-> deterministic WorkflowGenerationPlan / Workflow JSON generator
```

The provider may improve summary, missing questions, service extraction, FAQ extraction, policies, and product catalog draft fields. It must not produce the final WorkflowDefinition. Workflow JSON must still be generated by the existing deterministic generator and validated by `validateWorkflow()`.

## Provider Boundary

The approved implementation task should add an interface-compatible provider adapter under `packages/ai-builder-orchestrator` or a narrow adjacent package. The adapter should:

- Accept source text and the prompt pack.
- Return unknown provider output to the existing structured parser.
- Never expose raw provider response in UI, traces, or logs.
- Redact secret-like values before diagnostics.
- Be disabled by default.
- Be unavailable in CI unless tests use mocks.

## Secret Handling Policy

- Use backend-only environment variables or an ignored local config file only after explicit implementation approval.
- Never read `.flowai.local.json` from browser code.
- Never send API keys through request bodies.
- Never embed keys in workflow JSON, BusinessUnderstanding, SourceDocument, traces, logs, screenshots, docs, or tests.
- Add a preflight diagnostic that reports `configured: true/false` without printing the key.
- Keep provider errors sanitized.

## Model Selection Policy

- Do not hardcode a single permanent model as an architectural decision.
- Store model choice in backend-only configuration.
- Prefer a high-quality current OpenAI model for extraction quality when the implementation task starts.
- Re-check official OpenAI API docs during implementation before selecting the model.
- Allow task-specific model routing later, for example:
  - business extraction: highest quality configured model;
  - critic/safety: high-reliability configured model;
  - cheap fallback or tests: mocked provider only.

## Structured Output Requirements

Provider output must be strict JSON data matching the existing orchestrator boundary:

- `businessUnderstandingPatch`
- `productCatalogDraft`

The output must include source-backed evidence for factual claims. Product price, availability, eligibility, and recommendation-like fields must remain review-required unless sourceRefs exist.

Invalid, malformed, overbroad, unsupported, or unsafe output must fall back to deterministic behavior and surface a safe diagnostic.

## Prompt Rules

Prompts must require:

- Business owner friendly behavior.
- Arabic response support when source/user language is Arabic.
- One useful follow-up question at a time.
- No invented services, prices, availability, policies, or recommendations.
- No final Workflow JSON generation.
- No executable code or workflow expressions.
- Strict structured output only.
- SourceRefs for factual claims.

## Testing Plan

Required automated tests:

- Provider disabled path still returns deterministic fallback.
- Mock provider success path returns structured BusinessUnderstanding refinement.
- Malformed provider output falls back.
- Secret-like provider output is redacted.
- Product catalog claims without sourceRefs are blocked.
- Provider diagnostics never contain raw keys or raw response bodies.
- Studio still labels live AI as disabled unless explicitly configured.

Non-CI smoke test after implementation approval:

- Run with a local ignored key/config only.
- Use a small clinic or service markdown source.
- Confirm provider is called.
- Confirm output stays structured.
- Confirm no secrets appear in stdout, UI payload, workflow JSON, traces, or generated docs.
- Confirm deterministic fallback remains available when config is absent.

- AI can overclaim facts without sourceRefs if validation is weakened.
- The owner may mistake live extraction for publish-ready chatbot behavior.
- Model names and API recommendations can change; implementation must check official docs at that time.
